The `unaltredocs` profile can document more than software packages. Use it for datasets, models, dashboards, protocols, public APIs and reproducible project outputs.

<h2 id="project-resources-and-outputs">Project resources and outputs</h2>

Good output documentation usually includes:

- What the output is and who should use it.
- Input requirements and validation rules.
- Output files, schemas and expected formats.
- Reproducibility notes, including software versions and random seeds.
- Licenses, citation metadata and persistent identifiers.

Keep long reference material in documentation pages and use project/output collection pages for public-facing summaries.
